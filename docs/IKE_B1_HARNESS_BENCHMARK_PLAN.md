# IKE B1 Benchmark Plan

## Purpose

This document turns the `Harness / OpenClaw / AI agent` benchmark into the
next concrete milestone after the internal IKE v0.1 loop proof.

Reference:

- `docs/IKE_REAL_WORLD_BENCHMARKS.md`

## Benchmark Statement

IKE should be able to show, in a user-understandable way:

1. what changed in the world
2. which people / projects / communities matter
3. what the new concept means
4. why it matters to this project
5. what bounded next action should follow

## Why This Is The Next Milestone

The current internal loop proof is necessary, but the current visible
workspace is still too inspect-oriented and not meaning-oriented.

Therefore the next milestone should not be:

- more inspect controls
- more preview endpoints
- a generic trend platform

It should be:

- one concrete, reality-based benchmark proof

## Chosen Benchmark

The current benchmark is:

- `harness`

within the existing attention neighborhood:

- `openclaw`
- `AI agent`
- related evaluation / testing / runtime topics

## Slice Order

### B1-S1 Information Brain Slice

Goal:

- detect the rise or prominence of `harness`
- identify related entities:
  - people
  - projects
  - organizations
  - communities
- rank them by signal value

Output:

- one structured trend bundle for `harness`

This is the first required slice because the later slices depend on it.

### B1-S2 Knowledge Brain Slice

Goal:

- explain what `harness` means in context
- distinguish it from adjacent methods
- connect it to nearby concepts and projects

Output:

- one concept-and-relations summary derived from the trend bundle

### B1-S3 Evolution Brain Slice

Goal:

- decide why the trend matters to this project
- recommend one bounded next action

Output:

- one project-relevance judgment
- one bounded proposed action

## First Visible Milestone

The first visible milestone should be one user-readable insight card or report
that answers:

1. What changed?
2. Who matters?
3. What does it mean?
4. Why does it matter here?
5. What should we do next?

## Current Assessment

B1 has now produced a real user-readable benchmark report from live discovery
inputs.

That means B1 should now be treated as:

- `signal + meaning + relevance hint`

and not over-claimed as:

- a full research trigger
- a full evolution action engine
- a full concept-adoption engine

In practical terms:

- B1 proves that the benchmark direction is real
- B1 does not finish the benchmark family

## What B1 Still Does Not Fully Solve

Even after the first live report, B1 still remains weaker than the intended
end-state because it does not yet reliably:

- distinguish `concept-defining` vs `ecosystem-relevant` vs `implementation-relevant` entities
- produce deeper concept-boundary comparison
- escalate from relevance hint into a true research/prototype trigger
- verify its own findings against stronger second-pass evidence

## Transition To B2

The next stage should not be more ranking polish.

The next stage should be:

- `B2 Concept Deepening + Research Trigger`

That stage should deepen concept understanding, map the concept to explicit
mainline gaps, and upgrade the next action from low-grade observation into a
bounded research/prototype trigger when justified.

## Guardrails

Do not do these yet:

- no generic trend framework
- no public stable object store
- no broad new dashboard
- no fake confidence that all trend detection problems are solved

## Immediate Delegation Target

The next bounded packet should be:

- `B1-S1 Harness Trend Detection and Entity Ranking`

That packet should focus only on:

- current data inputs
- current source-intelligence outputs
- ranked signal-bearing entities for `harness`

It should not try to solve the whole benchmark in one patch.
