# 研发方法与质量保障体系

> 目标：明确 MyAttention 后续研发的统一工作方式，包括方案确认、设计先行、编码约束、测试门槛、自我进化闭环、非功能设计要求、阶段演进路径与交付标准。

---

## 1. 方法总则

### 1.1 核心原则

1. 主线优先，不做无主线支撑的零散功能。
2. 方案先行，未经确认不进入中大型编码。
3. 设计先行，编码必须有文档依据。
4. 事实优先，判断基于日志、测试、指标和真实运行结果。
5. 质量内建，测试和可观测性不是收尾工作。
6. 演进式实现，每一阶段都要可运行、可验证、可回退。
7. 主动研究与纠偏，不能机械附和需求或假设；应评估可行性、合理性、替代路线与潜在错误。
8. 研发过程是循环反馈优化过程，不按线性瀑布假设推进；遇到关键分歧、风险或证据冲突时，应及时回到讨论与校准。

### 1.2 适用范围

适用于：

- 信息大脑
- 知识大脑
- 进化大脑
- 底层架构、存储层、部署层
- API、前端、数据模型、调度、测试与运维能力

---

## 2. 标准研发流程

### 2.1 强制步骤

1. 确认任务是否服务于项目主线。
2. 明确问题、目标、边界、风险与验收标准。
3. 对需求和假设做研究、质疑和替代方案评估，必要时先纠偏再继续。
4. 产出方案或补充现有方案。
5. 落设计文档并完成评审确认。
6. 再开始编码。
7. 编码后完成相应层级测试与人工验证。
8. 更新实现文档、进度文档、已知问题与后续计划。
9. 将有效方法、失败方法、重要决策和过程产物沉淀入仓库，确保可追溯与可复制。

### 2.2 任务分级

#### L0：文档或纯说明改动

- 可直接修改文档
- 需要做一致性检查

#### L1：小范围逻辑改动

- 可在现有设计基础上编码
- 必须补最小验证

#### L2：模块级改动

- 必须先补设计文档
- 必须补自动化测试或回归验证

#### L3：跨模块或架构改动

- 必须先确认专项方案
- 必须形成专项设计文档
- 必须明确迁移路径、兼容策略、回退策略

## 3. 文档驱动开发要求

### 3.1 编码前必须明确的内容

- 要解决的核心问题
- 不解决什么
- 当前实现边界
- 涉及的模块
- 数据流变化
- 存储变化
- 接口变化
- 测试范围
- 风险和回退方式

### 3.2 每次中大型改动至少要有的产物

- 设计文档
- 实施计划
- 验收标准
- 进度记录

### 3.3 文档优先级

1. `PROJECT_MASTER_PLAN.md`
2. 专项架构设计文档
3. `SPEC.md`
4. 实现与测试文档
5. 进度记录

如果文档之间冲突，以更高优先级文档为准，并尽快修正文档体系。

### 3.4 变更与版本记录要求

- 任何 L2/L3 改动都必须更新项目级变更记录
- 任何架构、数据模型、兼容性调整都必须有版本化说明
- 任何自我进化策略调整都必须记录前后差异、原因和验证结果
- 详细规则见：`docs/CHANGE_MANAGEMENT.md`
- 统一版本分层与内部对象版本规则见：`docs/VERSION_MANAGEMENT.md`

### 3.5 外部 skill 使用原则

- 外部 skill 只能作为辅助，不替代项目主文档和真实验证
- 引入外部 skill 时，必须先明确适用环节、评估标准和停用条件
- 任何外部 skill 的试用效果都必须持续记录
- gstack 试用规则见：`docs/GSTACK_TRIAL_PLAN.md`
- 评估记录见：`docs/SKILL_EVALUATION_LOG.md`

## 4. 测试方法

### 4.1 测试原则

- 没有验证结果的改动，不视为完成
- 测试范围与改动级别匹配
- 优先保证核心链路稳定，而不是堆测试数量
- 测试结果必须可追踪、可归档、可被进化系统消费

### 4.2 测试层级

1. 单元测试
2. 集成测试
3. API 回归测试
4. 前端关键页面测试
5. E2E 测试
6. 迁移与数据兼容验证
7. 运行时健康检查
8. 性能与稳定性验证

### 4.3 改动类型对应的最低测试要求

#### 文档改动

- 文档一致性检查

#### 后端逻辑改动

- 至少一个直接相关测试或最小验证脚本
- 至少一次接口级验证

#### 前端改动

- 页面加载验证
- 关键交互验证

#### 数据模型或迁移改动

- 迁移执行验证
- 兼容或回滚说明

#### 架构级改动

- 分阶段验证结果
- 回退方案验证

### 4.4 测试输出要求

每次交付至少记录：

- 执行了什么测试
- 哪些通过
- 哪些未覆盖
- 剩余风险是什么

## 5. 自我进化闭环

### 5.1 定位

自我进化系统不是“以后再做”的增强项，而是项目核心基础设施。它负责把运行问题、测试问题、性能问题和重复问题沉淀成可处理的任务闭环。

补充要求：

- 进化大脑的主控 agent 必须具备研究、质疑、纠偏和替代路线建议能力。
- 它不能只执行已有规则，也不能只附和用户或上游代理的想法。
- 它应持续比较：
  - 当前方法是否仍然有效
  - 是否已有更好的方法、工具、skill、agent team 模式
  - 当前路线是否偏离项目总目标

### 5.2 最小闭环

```text
日志 / 测试 / 巡检结果
  -> 问题识别
  -> 去重归并
  -> 问题分级
  -> 生成或更新任务
  -> 修复
  -> 回归验证
  -> 状态归档
```

### 5.3 必须长期具备的能力

- 实时或准实时日志监控
- API 周期巡检
- 关键页面周期巡检
- 测试结果归档
- 重复问题去重归并
- 问题历史追踪
- 失败回归重试
- 质量趋势统计

### 5.4 进化路径

#### Phase A：可见性

- 有日志
- 有状态面板
- 有健康检查

#### Phase B：自动发现

- 自动识别异常日志
- 自动发现测试失败
- 自动生成问题记录

#### Phase C：问题闭环

- 重复问题归并
- 问题升级、降级、关闭
- 问题与代码/模块关联

#### Phase D：质量驱动优化

- 基于问题密度和失败率指导优先级
- 基于指标自动发现薄弱模块
- 基于回归结果评估改动质量

#### Phase E：策略级自适应

- 根据质量反馈调整抓取策略、测试范围和告警阈值
- 为后续自动化修复提供安全边界

## 6. 非功能设计要求

### 6.1 可维护性

- 模块职责明确
- 存储边界清晰
- 配置集中管理
- 文档和实现保持同步

### 6.2 可测试性

- 核心逻辑尽量纯函数化或可注入依赖
- 外部服务必须可 Mock
- 关键链路必须有可重复验证方式

### 6.3 可观测性

- 关键服务必须有日志
- 关键链路必须有指标
- 关键任务必须有状态
- 错误必须可追溯到模块和输入

### 6.4 可扩展性

- 单机不是架构前提
- 存储层和处理层要支持后续拆分
- 原始层、事实层、富化层、聚合层必须分离

### 6.5 性能

- 常用查询要有明确优化路径
- 热路径优先缓存而不是提前复杂化
- 聚合分析优先离线或准实时任务

### 6.6 稳定性

- 任务必须可重试
- 操作必须尽量幂等
- 关键链路要有降级或回退策略

### 6.7 安全性

- API Key 与敏感配置不写死
- 日志中避免泄露敏感信息
- 外部集成必须有失败隔离

### 6.8 可部署性

- 本地开发、测试、生产环境职责分明
- 环境差异尽量通过配置收敛
- 后续多机部署不要求改业务逻辑

## 7. 阶段演进要求

### 7.1 当前阶段重点

1. 完成信息流事实层与存储层重构
2. 完善自我进化与测试闭环
3. 建立知识大脑的数据结构基础

### 7.2 阶段完成的判断标准

一个阶段只有同时满足以下条件才算完成：

- 有设计文档
- 有实现结果
- 有测试证据
- 有进度记录
- 有剩余风险说明

### 7.3 不允许的情况

- 先写代码后补设计
- 没有测试直接宣布完成
- 只在内存里跑通却当成长期能力
- 没有日志和状态就引入后台能力

## 8. 交付与复盘要求

### 8.1 每次交付必须说明

- 做了什么
- 为什么这样做
- 如何验证
- 当前限制是什么
- 下一步是什么

### 8.2 每个阶段结束必须复盘

- 主线是否清晰
- 设计是否与实现偏离
- 当前最大风险在哪里
- 下一阶段是否需要先补设计

## 10. Encoding And I18N

- All source code, config, SQL, JSON, Markdown, TOML, and script templates must use UTF-8.
- Business code must not rely on OS default encodings, especially Windows console encoding.
- New frontend UI copy must go through centralized i18n resources instead of ad hoc hardcoded strings.
- Mojibake, replacement characters, and question-mark fallback are functional defects, not cosmetic defects.
- Encoding consistency checks and real UI inspections must be part of automated testing and the evolution loop.
- Detailed policy: [ENCODING_AND_I18N_STANDARD.md](./ENCODING_AND_I18N_STANDARD.md)
