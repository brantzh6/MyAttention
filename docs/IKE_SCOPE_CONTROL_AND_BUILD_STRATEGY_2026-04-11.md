# IKE Scope Control And Build Strategy

Date: 2026-04-11
Status: controller baseline

## Purpose

Define what IKE should build itself, what it should adapt from external agent
systems, and what should be explicitly de-scoped or delayed.

This document exists because the project has now been running for more than a
month, and the current strategic risk is no longer "lack of ideas".

The current risk is:

- too many important lines
- too many adjacent hard problems
- too much temptation to build every layer ourselves

## Core Problem

IKE is touching multiple problem spaces that are each large enough to become an
independent project:

- runtime
- task management
- memory management
- harness
- source intelligence
- evolution brain
- knowledge organization
- cross-disciplinary discovery

If all of these are treated as first-class self-built platforms at the same
time, the project will likely become architecturally rich but executionally
weak.

## Core Strategic Rule

IKE must build the smallest self-owned core that preserves:

- judgment
- truth
- governance
- absorptive intelligence
- AI-driven evolution capability

IKE must not try to reimplement every execution substrate around that core.

## Three-Layer Build Model

### Layer 1. Self-owned core

These are IKE-native and should remain under direct controller ownership:

- controller judgment
- runtime canonical truth for project / task / decision / work context /
  memory packet
- review / promotion / rollback governance
- source intelligence judgment logic
- external-method absorption logic
- long-term direction of evolution brain

Rule:

- these are not delegated as final authority
- they may be implemented with help from delegates
- but acceptance and semantics stay inside IKE

### Layer 2. Adapted execution layer

These may rely on external systems, but only behind IKE contracts:

- Claude Code
- Hermes-like skills and orchestration patterns
- OpenClaw
- qoder
- vector stores or retrieval substrates
- session / tmux / worktree execution surfaces

Rule:

- external systems are allowed to execute work
- external systems are not allowed to become IKE's canonical truth or final
  controller

### Layer 3. Deferred platform ambitions

These should not be treated as current full self-build programs:

- general-purpose harness platform
- general-purpose task board product
- general-purpose memory OS
- complete knowledge graph / world-model platform
- full autonomous multi-agent OS

Rule:

- keep them as future directions or narrow packets
- do not let them silently expand into immediate build scope

## What IKE Should Build Now

### 1. Runtime kernel

Why:

- this is the future home for durable task / decision / work context truth
- this reduces dependence on chat-memory continuation

### 2. Source Intelligence M1

Why:

- source quality is a direct product bottleneck
- this is more product-proximate than further broad harness ambition

### 3. Governance and rollback shell

Why:

- without this, self-evolution and delegated acceleration are unsafe

### 4. External method absorption lane

Why:

- the project cannot afford to rediscover every good external method itself
- but it also cannot blindly copy external systems

## Multi-Model Judgment Rule

Multi-model review / judgment should be incubated in the project first, but it
should not be designed as permanent project-private business logic.

Preferred pattern:

- incubate locally
- keep inspect-first truth boundaries
- design semantics so the capability can later be reused by other brains,
  review lanes, or agents

## What IKE Should Reuse Or Adapt

### Claude Code

Use for:

- bounded coding
- review
- testing
- long-running implementation sessions when needed

Do not treat as:

- IKE controller
- IKE canonical memory
- IKE final truth source

### Hermes

Use as:

- a method reference
- a delegation/orchestration reference
- a skill-system reference
- a candidate interviewable external agent

Do not treat as:

- a platform to copy wholesale
- a substitute for IKE runtime truth and governance

### External vector / retrieval / storage systems

Use as:

- substrate

Do not treat as:

- the same thing as reviewed knowledge truth

## What Should Be Explicitly Compressed Or Deferred

### 1. Full harness redesign

Current rule:

- keep hardening narrow
- do not let harness become the whole project

### 2. Full task-management product

Current rule:

- use durable task surfaces
- do not try to build a universal project-management platform now

### 3. Full knowledge-universe architecture

Current rule:

- preserve the north-star
- implement only the next narrow truthful packets

## Current Overexpansion Signals

The controller should treat these as warning signs:

- a support track starts behaving like a co-equal mainline
- a research line keeps producing concepts without bounded packets
- documents grow faster than implementation and validation
- more systems are described than actually run
- external systems are studied without a clear absorption judgment

## Scope Compression Rules

When a line expands too far, compress it by asking:

1. Is this part of the self-owned core?
2. Can this be adapted from external systems?
3. Is this needed for the next milestone, or only strategically interesting?
4. Can the next move be reduced to one bounded packet?

If the answer to 1 is no and 3 is no, it should usually not be an active build
line.

## Current Controller Conclusion

IKE should be treated as:

- a controller and runtime kernel project
- with external execution substrates
- plus a future knowledge and source-intelligence expansion path

IKE should **not** currently be treated as:

- a full self-built agent OS
- a full self-built universal knowledge platform
- a full self-built general harness platform

## Recommendation

`accept_with_changes`

Reason:

- the current direction is still salvageable and strategically strong
- but scope must now be actively controlled, not only discussed
