# 第一批受控外包任务候选

## 1. 目的

本文件列出当前最适合通过受控外包方式拆出去的子任务。

筛选标准：

- token 消耗高
- 边界清晰
- 可独立验证
- 失败可回收
- 不侵入主控决策

## 2. P0 候选

### D-001 信息流首页 freshness 修复

- 类型：coding-heavy
- 目标：
  - 让首页明确显示数据新鲜度
  - 避免本地缓存先展示造成“系统没抓到新数据”的错觉
- 预期输出：
  - 前端补丁
  - freshness 状态说明
  - 验证结果
- 验收：
  - 页面可明确看到最后刷新时间/数据时间
  - 不再默认让旧缓存伪装成最新状态

### D-002 Source discovery 质量评估报告

- 类型：bounded research / review
- 目标：
  - 分析当前建源结果为什么质量不高
  - 识别 domain 思维、对象类型缺失、配额不平衡等问题
- 预期输出：
  - 结构化问题清单
  - 质量改进建议
  - 候选发现对象类型清单
- 验收：
  - 能直接转成下一步实现任务

### D-003 Source discovery adapter 原型

- 类型：coding-heavy
- 目标：
  - 为对象发现适配层实现第一版可插拔接口/原型
- 预期输出：
  - adapter 接口草稿
  - 至少一个对象发现原型
  - 测试或烟测
- 验收：
  - 不改变主控策略前提下，可接入现有 source intelligence

### D-004 Evolution 页面问题中心可视化整理

- 类型：UI/review-heavy
- 目标：
  - 让问题、事件、产物、建议动作更容易理解
- 预期输出：
  - 页面层级改进
  - 状态文案/标签优化
  - 验证截图或说明
- 验收：
  - 用户能更快看懂发生了什么、下一步是什么

## 3. P1 候选

### D-101 Feed Redis 缓存接入方案和原型

- 类型：coding-heavy
- 目标：
  - 把 feed 主热缓存从纯内存推进到 Redis
- 预期输出：
  - 接入方案
  - 原型补丁
  - 验证结果

### D-102 UI 视觉巡检批处理

- 类型：review-heavy
- 目标：
  - 对首页、聊天页、evolution 页做视觉一致性巡检
- 预期输出：
  - 问题列表
  - 修复建议
  - 必要时补丁

### D-103 Source intelligence 分类与展示重构草案

- 类型：design/review-heavy
- 目标：
  - 让 topic、plan、candidate、object type 的层级更清楚
- 预期输出：
  - 页面结构建议
  - 信息层级建议
  - 文案/标签方案

## 4. 暂不外包

- 主线优先级和阶段切换判断
- 认知框架/方法框架最终定稿
- 受控外包策略本身的验收
- 主控脑与多脑协作策略最终裁决
- 高风险迁移或会破坏数据底座的任务

## 5. 执行顺序建议

1. `D-002` 先做质量分析
2. `D-001` 修 freshness 感知
3. `D-003` 做对象发现适配层原型
4. `D-004` 整理 evolution 可视化
5. 再进入 `D-101`

## 6. 说明

本清单是 `V1`，后续会随着：

- token 消耗统计
- 外包成功率
- 验收成本
- 回滚成本

持续更新。
