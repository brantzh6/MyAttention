# IKE Vision, Design, Architecture, And Path Alignment

Date: 2026-04-17
Status: controller baseline

## Purpose

Clarify the relationship between:

- the unchanged project vision
- the current design expression
- the architecture being built
- the implementation path, which may change over time

This document exists to prevent a recurring drift:

- local implementation packets get mistaken for the project itself
- runtime gets mistaken for the final product identity
- bounded heuristics get mistaken for the long-term intelligence model

## Core Judgment

The project's main objective has **not** changed.

IKE is still being built as an **AI-driven evolution system**.

What can change is the path used to reach that objective.

That means:

1. the vision is stable
2. the design expression may be refined
3. the architecture may be staged
4. the implementation path may be adjusted when a better route is found

## 1. Vision Layer

The stable vision is:

- build an AI-driven system that can continuously improve how it discovers,
  organizes, evaluates, absorbs, and evolves knowledge work
- preserve a living relationship between:
  - information brain
  - knowledge brain
  - evolution brain
  - world model
  - thinking tools / scientific methodology

This is not a static workflow engine.

This is not a fixed rules project.

This is not a runtime-first product identity.

## 2. Design Expression Layer

The current design expression should be read as a layered model:

1. information brain
2. knowledge brain
3. evolution brain

Cross-cutting all three:

- world model
- thinking tools / scientific methodology

Interpretation:

- information brain handles discovery, intake, signal triage, and current-flow
  awareness
- knowledge brain handles reviewed absorption, synthesis, memory shaping, and
  canonical/provisional separation
- evolution brain handles diagnosis, prioritization, adaptation, and what to
  improve next
- world model provides the larger explanatory structure that connects objects,
  systems, disciplines, and context
- thinking tools / scientific methodology provide the operational method layer:
  hypothesis, comparison, experiment design, error analysis, and iterative
  refinement

So the brains are **not** flat siblings.

They are layered, dependent, and expected to evolve.

## 3. Architecture Layer

The architecture currently being built is a staged support architecture for the
vision above.

Its job is to make the vision executable in bounded steps.

Current architectural interpretation:

- `Runtime v0` is a dependency component and accepted substrate
- `Source Intelligence V1` is the current active capability line above that
  substrate
- governance, review, rollback, and alignment shells exist to keep the system
  truthful while it is still incomplete
- bounded heuristics and inspect-only AI judgment lanes are scaffolds, not the
  final intelligence architecture

Important rule:

Runtime is part of the architecture.

Runtime is **not** the project's final identity.

## 4. Implementation Path Layer

The implementation path is allowed to change.

Examples of legitimate path changes:

- choosing runtime first to establish durable task/decision truth
- moving from cleanup heuristics to AI-assisted judgment once input quality is
  sufficient
- adapting external execution systems instead of building every substrate
  internally
- reordering bounded packets when the current path exposes a clearer bottleneck

Examples of illegitimate path drift:

- treating local cleanup heuristics as the end-state intelligence model
- treating runtime completion as the completion of IKE
- letting support infrastructure replace the actual AI-evolution objective
- forgetting the world-model / methodology layer because near-term packets are
  easier to code

## 5. Practical Controller Rule

When judging any active work line, ask:

1. does it serve the unchanged vision?
2. is it part of the current staged architecture?
3. is it only a path choice, or is it incorrectly redefining the project?

If a work line only makes sense as a path choice, it must not be narrated as
the new project identity.

## 6. Current Project Reading

The current truthful reading is:

- the project goal remains an AI-driven evolution system
- the current design expression is layered brains plus cross-cutting world
  model and thinking tools
- the current architecture uses runtime as substrate, not as the end-state
  product identity
- the active implementation path is now primarily through `Source Intelligence`
  and AI-assisted judgment, not through further runtime-first expansion

## 7. Non-Forget Rule

Any future document that describes current mainline or near-term execution
should preserve all four layers:

1. stable vision
2. current design expression
3. staged architecture
4. adjustable implementation path

If one of these four disappears, the project narrative is incomplete.
