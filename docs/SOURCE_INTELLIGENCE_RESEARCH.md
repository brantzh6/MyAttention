# 来源智能研究

> 目标：在真正改造信息流之前，先研究 MyAttention 应如何从“固定 RSS 列表”升级为“目标驱动、冷热分层、可持续校准”的来源智能体系。

---

## 1. 为什么短期要先做这一块

当前短期主线如果继续深挖：

- 多大脑
- 复杂协作
- 长期进化

但信息抓取基础仍然停留在“固定源 + 固定抓取”，会带来两个直接问题：

1. 后续研究和进化用到的信息样本会持续偏
2. 方案判断容易建立在过时、狭窄、失衡的信息入口上

因此，短期优先级上，信息抓取建设是基础工程。

但这里需要纠偏：

- “信息抓取建设”不等于先去加更多 RSS
- 真正应该补的是来源智能与获取策略层

---

## 2. 核心区分

### 2.1 来源发现 vs 来源执行

这是当前最关键的结构性区分。

#### 来源发现

回答：

- 该关注什么对象？
- 为什么关注？
- 它对当前目标的价值是什么？
- 它是权威源、信号源、前沿源还是社区源？

对象可能是：

- 机构
- 作者
- 社区
- 账号
- 仓库
- 站点
- 论文库
- 标准组织
- 话题 / 标签

#### 来源执行

回答：

- 用什么方式拿数据？
- 多久拿一次？
- 是持续抓取，还是低频 review，还是一次性导入？

执行手段可能是：

- RSS
- API
- 普通 fetch
- 浏览器抓取
- agent-assisted retrieval
- 按需搜索

结论：

- 这两层必须分开设计
- 否则会把“关注对象”和“抓取手段”混成一个 `url`

---

### 2.2 冷数据、温数据、热数据

不是所有来源都应该进入持续高频信息流。

#### 冷数据

特点：

- 长期稳定
- 变化极慢
- 更适合一次性导入或非常低频 review

例子：

- 百年前的历史事件资料
- 经典论文
- 基础理论
- 固定版本标准

策略：

- 一次性采集
- 结构化整理
- 低频人工/自动 review

#### 温数据

特点：

- 不是天天变化
- 但会有版本更新或理解更新

例子：

- 综述论文
- 学会指南
- 官方白皮书
- 标准修订

策略：

- 低频 review
- 版本比较
- 变化检测

#### 热数据

特点：

- 持续变化
- 对前沿研究、决策、趋势判断有强时效要求

例子：

- GitHub 项目动态
- 研究社区动态
- 社交平台讨论
- 新闻与市场信息

策略：

- 高频监控
- 去噪
- 快速沉淀事实与信号

---

## 3. 来源类型不应只等于技术协议

当前系统中更容易出现的误区是：

- RSS = 一类来源
- API = 一类来源
- WEB = 一类来源

但这是执行协议，不是来源本质。

后续更合理的来源性质应包括：

- `authority source`
- `signal source`
- `frontier source`
- `community source`
- `stable knowledge source`
- `reviewable source`
- `live intelligence source`

而技术协议只是执行层字段。

---

## 4. 短期最值得建设的能力

### 4.1 主题驱动建源

输入：

- 一个主题
- 一个目标
- 一个关注类型（权威 / 前沿 / 最新 / 方法）

输出：

- 候选来源集合
- 候选对象集合（机构、作者、社区、仓库、账号）
- 推荐执行策略

这比继续人工往 `feed_sources.json` 填列表更重要。

---

### 4.2 来源策略分层

后续应能对每个来源明确：

- 是否持续跟踪
- 跟踪频率
- 执行方式
- 是否需要代理
- 是否需要浏览器/反爬
- 是否只做一次性获取
- 是否只做低频 review

也就是说，来源对象后面至少要有：

- `source_nature`
- `monitoring_mode`
- `review_interval`
- `execution_strategy`
- `authority_confidence`
- `target_use_case`

---

### 4.3 来源更新与淘汰

来源管理不是只增不减。

后续至少应支持：

- `promotion`
- `demotion`
- `replacement`
- `retirement`
- `rediscovery`

否则信息流只会越来越脏、越来越重、越来越保守。

---

## 5. agent-assisted retrieval 的位置

这里需要明确边界。

### 不适合让 agent 全面接管的地方

- 全量持续抓取
- 高频批量拉取
- 事实层长期主存储入口

原因：

- 成本高
- 稳定性差
- 可回放性弱
- 很难长期规模化

### 适合让 agent 参与的地方

- 建源探索
- 新主题的入口发现
- 复杂站点/复杂反爬场景
- 特定一次性研究任务
- 对信号源做定向探索

因此，后续应把 agent 放在：

- `source discovery`
- `special retrieval`
- `strategy exploration`

而不是直接替代 collector。

---

## 6. 短期实施建议

在不引爆范围的前提下，短期最合理的实施顺序是：

1. 先补 `source discovery` 能力
2. 再补来源性质与策略字段
3. 再补 review / refresh / retirement 机制
4. 最后再考虑哪些来源需要 agent-assisted retrieval

这比一开始就重写整个抓取器稳得多。

---

## 7. 研究结论

1. 短期主线应优先建设“来源智能”，而不是继续堆固定抓取源。
2. 来源发现与来源执行必须分开设计。
3. 信息对象必须按冷热和变化频率区分，而不是一律持续抓取。
4. 来源本质应先按性质分类，再决定执行协议。
5. agent-assisted retrieval 是重要能力，但应放在建源和复杂获取位置，不应直接替代持续采集层。
