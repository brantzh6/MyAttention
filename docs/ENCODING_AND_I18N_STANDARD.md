# 编码与国际化工程规范

## 目标

统一源码、配置、脚本、文档和 UI 文案的编码规则，避免 Windows、macOS、Linux 混合环境下再次出现乱码、构建异常、页面错乱和误判问题。

## 强制规则

1. 所有源码、配置、SQL、JSON、Markdown、TOML、YAML、脚本模板统一使用 `UTF-8`。
2. 不得依赖操作系统默认编码，不得把终端显示结果当成文件真实编码。
3. 手工代码编辑只允许使用 `apply_patch`。
4. 批量生成或程序化改写文本文件时，只允许使用显式指定 `UTF-8` 的 Python 脚本。
5. 禁止使用 PowerShell 重定向、here-string、`Set-Content`、`Out-File` 等方式写入项目文本文件内容。
6. 前端新增文案不得继续散落硬编码，必须逐步进入 i18n 资源层。
7. 任何乱码、`�`、问号替换、mojibake、构建期编码异常，都按功能缺陷处理。

## 文件写入约定

### 允许方式

- `apply_patch`：手工修改源码、配置、文档。
- Python 脚本：批量生成、批量迁移、结构化文本改写。

### Python 脚本要求

- 必须显式使用 `encoding="utf-8"`。
- 读取和写入都必须指定编码。
- 不允许依赖默认编码。

示例：

```python
from pathlib import Path

path = Path("example.txt")
text = path.read_text(encoding="utf-8")
path.write_text(text + "\n", encoding="utf-8")
```

### 禁止方式

- PowerShell `>`、`>>`
- PowerShell here-string 直接落文件
- `Set-Content`
- `Out-File`
- 任何未显式指定 UTF-8 的文件写入方式

## 前端文案与 i18n

1. 前端页面结构文案必须进入统一字典或 locale 资源。
2. 至少支持 `zh-CN` 和 `en` 两套 UI 框架文案。
3. 模型原始回答可以保留原语言，但页面结构文案必须可切换。
4. 高风险链路允许临时使用 Unicode 转义过渡，但最终文件仍必须是 UTF-8。

## 存储与运行时要求

1. 数据库存储必须优先使用 UTF-8 编码实例。
2. 若存在历史编码遗留，系统任务标题、系统历史记录、内部消息应优先采用 ASCII-safe 文本，避免再次阻塞系统闭环。
3. 日志优先结构化，不把大段自然语言日志作为唯一事实源。
4. 进化大脑必须包含真实 UI 巡检，不得只看日志和接口返回码。

## 测试要求

以下改动必须做编码相关验证：

- 前端页面改动
- UI 文案改动
- 文件读写逻辑改动
- 导入/抓取链路改动
- Windows 本机运行脚本改动

至少应验证：

- 页面文本是否正常渲染
- API 返回内容是否正常
- 关键页面是否存在乱码和样式异常
- 文件是否保持 UTF-8

## 当前执行约定

从现在开始：

- 我不再使用 PowerShell 直接写项目文本文件。
- 手工修改一律使用 `apply_patch`。
- 需要脚本化写文件时，只使用显式 `UTF-8` 的 Python。
- 遇到编码问题，优先当成系统设计问题处理，而不是显示层小问题。
