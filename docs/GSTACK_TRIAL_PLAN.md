# gstack Skill 试用与评估方案

> 目标：以受控试用的方式把 gstack 的设计、工程评审、QA、文档收尾能力纳入 MyAttention 研发流程，并持续评估它是否真的提升质量和效率。如果效果不及预期，及时缩减或停止使用。

---

## 1. 试用原则

1. gstack skill 是辅助能力，不是项目事实来源。
2. 项目总计划、专项设计文档、真实代码和真实测试结果，优先级高于 skill 输出。
3. 先小范围试用，再决定是否扩大使用。
4. 每次使用都要记录效果，避免凭感觉长期沿用。
5. 只在适合的环节使用，不把所有工作都外包给 skill。

---

## 2. 当前纳入试用的 skill

- `plan-eng-review`
- `plan-design-review`
- `design-consultation`
- `review`
- `qa`
- `document-release`

说明：

- 这些 skill 已安装到本机 Codex skills 目录
- 需要重启 Codex 后，后续会话才能正式使用

---

## 3. 在本项目中的推荐使用环节

### 3.1 `plan-eng-review`

适用：

- L2/L3 改动
- 架构调整
- 数据流重构
- 存储层改造
- 自我进化系统设计

用途：

- 补充架构风险审查
- 检查复杂度、边界、测试矩阵、失败路径

### 3.2 `plan-design-review`

适用：

- 新页面
- 关键设置页
- 信息流、知识库、聊天相关交互

用途：

- 在编码前发现空状态、层级、响应式、边界状态缺漏

### 3.3 `design-consultation`

适用：

- 需要建立或重做视觉系统时
- 需要形成统一 `DESIGN.md` 时

当前结论：

- 可以保留
- 但不作为当前主线高优先级工具

### 3.4 `review`

适用：

- 代码改动完成后
- 合并前
- 关键缺陷修复后

用途：

- 补充结构性风险、遗漏测试、兼容性问题检查

### 3.5 `qa`

适用：

- 前后端联调
- 关键页面回归
- 功能闭环验证

用途：

- 用真实交互发现缺陷
- 为修复后补回归验证

### 3.6 `document-release`

适用：

- 阶段性交付后
- 文档可能漂移的改动后

用途：

- 补齐 README、架构文档、CHANGELOG、TODO 等

---

## 4. 不适合使用的情况

- 纯后端小修复
- 纯文档改动
- 明确无 UI、无交互、无设计范围的小任务
- 已有清晰项目规则且 skill 会明显扩大范围的情况
- 时间很紧且评审成本高于收益的情况

---

## 5. 评估维度

每次使用后至少从以下维度评估：

1. 是否发现了我们原本没看到的重要问题
2. 是否提升了方案完整性
3. 是否改善了测试覆盖和回归质量
4. 是否减少了文档遗漏
5. 是否明显增加了无效流程或噪音
6. 是否出现过度扩 scope 的倾向
7. 是否与本项目既有规则冲突

---

## 6. 成功判定标准

以下情况说明试用有效：

- 连续多次帮助发现真实问题
- 对计划质量有明显提升
- 对测试范围和回归完整性有帮助
- 没有显著拖慢交付节奏
- 输出内容与本项目主线一致

---

## 7. 失败或停用条件

出现以下任一情况，应缩减或停止使用对应 skill：

- 连续多次只输出泛泛而谈的建议
- 经常无视项目现有设计和主线，导致范围膨胀
- 明显增加流程成本但没有产出真实价值
- 输出大量误报，干扰判断
- 经常与项目文档规则冲突
- 依赖链过重，使用门槛过高，实际无法稳定落地

停用策略：

- 先停止单个 skill
- 再评估是否保留其余 skill
- 必要时整体停止 gstack skill 试用

---

## 8. 使用规则

1. 使用前先确认它适合当前任务环节。
2. 使用后必须记录到 `docs/SKILL_EVALUATION_LOG.md`。
3. skill 输出不能直接覆盖项目设计结论，必须经过项目上下文校验。
4. 若 skill 建议与 `PROJECT_MASTER_PLAN.md` 或专项设计冲突，以项目文档为准。
5. 若 skill 明显扩大范围，应主动压缩回项目当前 Phase。

---

## 9. 评估记录方式

统一记录到：

- `docs/SKILL_EVALUATION_LOG.md`

每条记录至少包含：

- 日期
- 使用的 skill
- 用于什么任务
- 输出是否有帮助
- 发现了什么问题
- 是否值得继续使用

---

## 10. 当前试用结论

当前判断：

- 值得试用
- 但只建议优先试用以下 5 个：
  - `plan-eng-review`
  - `plan-design-review`
  - `review`
  - `qa`
  - `document-release`

保留但低优先级：

- `design-consultation`

原因：

- 当前项目主线还是架构、数据层、测试闭环
- 不是品牌级视觉重塑阶段
