# 项目推进路线图

> 目标：把已经完成部分能力的 MyAttention，从“能力散点 + 局部可用”推进到“围绕最终目标持续演进的完整系统”。本路线图强调阶段目标、与现有代码的对接点、交付物、验收标准和跟踪方式。

---

## 1. 先把当前状态说清楚

### 1.1 已经有的能力

当前项目已经具备一批可运行能力：

- 聊天与多模型对话
- 知识库与 RAG
- 记忆能力
- 信息源管理、代理控制、抓取基础能力
- 系统状态检查
- 自我进化基础能力
- 测试问题中心

这些能力说明项目不是从零开始。

### 1.2 当前真正的问题

问题不是“什么都没有”，而是：

- 长期目标已经很大，但底层数据与流程还没有完全统一
- 一部分能力仍停留在缓存、局部状态或点状实现
- 新架构方案已经明确，但还没有系统性接到现有代码上

所以，接下来的工作不是重写项目，而是：

**围绕新方案，把现有代码逐步接入新的主线架构。**

---

## 2. 当前总判断

现在可以认为：

- 项目总体目标已经重新明确
- 主架构和研发方法已经完成第一轮规划
- 可以进入“分阶段实施”阶段

这意味着当前工作重心已经从“继续讨论方向”切换为：

**按阶段改造现有项目，让现有能力逐步成为新架构的一部分。**

---

## 3. 当前阶段的核心主线

后续推进不以“页面数量”或“功能点数量”为主线，而以三条主线推进：

### 3.1 信息大脑主线

目标：

- 让信息流具备长期沉淀、分析、趋势、推荐和知识转化能力

当前最优先：

- 先解决存储和事实层

### 3.2 知识大脑主线

目标：

- 让知识不只是知识库文档堆积，而是有学科组织、权威理解、前沿研究、交叉关联

当前最优先：

- 先准备信息到知识的稳定输入

### 3.3 进化大脑主线

目标：

- 让系统能持续监控、测试、识别问题、归并问题、评估质量变化

当前最优先：

- 保持常驻运行
- 同时让后续每个 Phase 都可被它观测和验证

---

## 4. 项目推动总顺序

建议按以下顺序推进：

1. 打底信息大脑的数据底座
2. 把信息流主链路切到事实层
3. 完成信息富化与聚类
4. 建立趋势分析与深度总结能力
5. 再强化知识大脑结构化沉淀
6. 同步增强进化大脑的监控、评测和回归闭环

原因很明确：

- 没有稳定事实层，后面的趋势、总结、知识抽取和自我优化都不稳

---

## 5. 分阶段推进计划

## Phase 1：对象存储与原始层落地

### 目标

- 优化信息大脑的存储能力
- 建立原始层和对象存储抽象

### 要做什么

- `ObjectStore` 抽象
- `LocalObjectStore`
- `raw_ingest` 元数据表
- 最小原始层写入服务
- 选一个现有入口接入新链路

### 与现有代码怎么对接

优先对接现有：

- `services/api/feeds/fetcher.py`
- `services/api/routers/feeds.py`

但本阶段只选一个最小入口：

- 推荐先改 `/api/feeds/import`

原因：

- 风险低
- 更容易验证
- 不会一次性冲击现有抓取主链路

### 本阶段产出

- 存储抽象代码
- 数据迁移
- 原始层最小写入能力
- 测试和验证记录

### 验收标准

- 原始对象能写入对象层
- `raw_ingest` 有完整元数据
- 本地开发模式可运行
- 文档和变更记录同步完成

---

## Phase 2：信息流事实层落库

### 目标

- 把信息流从“缓存主导”切到“数据库事实层主导”

### 要做什么

- `feed_items` 成为主事实表
- 建立 `raw_ingest -> feed_items` 关联
- 将标准化条目稳定持久化
- 为 `/api/feeds` 切换读路径做双写准备

### 与现有代码怎么对接

重点对接：

- `services/api/feeds/fetcher.py`
- `services/api/routers/feeds.py`
- `services/api/db/models.py`

处理方式：

- 保留现有缓存
- 但缓存只做短期加速
- PostgreSQL 成为事实来源

### 本阶段产出

- `feed_items` 持久化主链路
- 兼容旧接口的过渡实现
- 接口级验证

### 验收标准

- 信息条目重启后仍存在
- `/api/feeds` 可以逐步切到事实层
- 缓存不再是主事实来源

---

## Phase 3：富化、聚类、embedding

### 目标

- 让信息流从“条目列表”进化为“可分析数据”

### 要做什么

- `feed_enrichments`
- 标签、主题、实体、可信度、重要度
- 聚类和事件簇
- embedding 与相似内容召回

### 与现有代码怎么对接

重点对接：

- `services/api/feeds/authority.py`
- `services/api/feeds/ai_brain.py`
- `services/api/feeds/evolution.py`
- `services/api/rag/engine.py`
- `services/api/knowledge/*`

### 本阶段产出

- 富化层表
- 聚类和相似性能力
- 更高质量的信息标注结果

### 验收标准

- 每条信息具备可用分析标签
- 事件簇可追溯
- 向量记录能回溯到事实层

---

## Phase 4：趋势分析与深度总结

### 目标

- 让信息大脑真正产生研究价值

### 要做什么

- 聚合日报/周报/月报
- 来源趋势变化
- 主题热度与热度迁移
- 深度总结与专题总结

### 与现有代码怎么对接

重点对接：

- `services/api/feeds/metrics.py`
- `services/api/feeds/evolution.py`
- 通知模块
- 前端信息流展示页

### 本阶段产出

- 聚合分析表
- 趋势接口
- 深度总结输出

### 验收标准

- 可查询趋势
- 可生成阶段性总结
- 可为推荐和研究提供数据输入

---

## Phase 5：知识大脑结构化升级

### 目标

- 让知识层从“知识库 + 向量检索”升级为结构化知识体系

### 要做什么

- 学科树和知识分类
- 权威来源类型
- 当前理解 / 权威共识 / 前沿研究 分层
- 跨学科关系组织

### 与现有代码怎么对接

重点对接：

- `services/api/knowledge/graph.py`
- `services/api/knowledge/*`
- `services/api/memory/*`
- 聊天和 RAG 路由

### 本阶段产出

- 知识结构模型
- 知识与信息的稳定转化链路
- 更强的问答与研究支持

### 验收标准

- 知识不再只是文档块
- 可以按学科和主题组织知识
- 可以把信息流成果沉淀成可复用知识

---

## Phase 6：进化大脑强化

### 目标

- 让整个系统的改动和质量变化都可被持续跟踪

### 要做什么

- API 回归任务体系
- 关键页面巡检
- 问题簇与问题历史
- 质量趋势和失败趋势分析
- 进化策略版本化与运行记录

### 与现有代码怎么对接

重点对接：

- `services/api/feeds/auto_evolution.py`
- `services/api/feeds/log_monitor.py`
- `services/api/routers/testing.py`
- `services/api/pipeline/log_analysis_scheduler.py`

### 本阶段产出

- 更强的自测闭环
- 质量趋势可视化
- 进化策略留痕和评估能力

### 验收标准

- 每个关键模块都有持续评估输入
- 重复问题能归并
- 每次重要改动都可被回归验证

---

## Phase 7：分布式部署准备

### 目标

- 为未来单机之外的扩展做准备

### 要做什么

- API / worker 分离
- 对象存储切换到 MinIO / S3 / OSS
- Redis、Qdrant、PostgreSQL 独立部署
- 多环境治理

### 与现有代码怎么对接

- 不重写业务逻辑
- 通过配置和部署层改造完成迁移

### 验收标准

- 单机代码逻辑不依赖单机文件路径
- 关键后台任务可拆成独立 worker

---

## 6. 每个阶段如何与现有代码对接

原则只有三个：

### 6.1 保留已有能力

- 不推倒重来
- 先保留现有聊天、知识库、信息流、进化能力

### 6.2 旁路接入新架构

- 新链路先并进来
- 旧链路暂时保留
- 通过双写、旁路、过渡读路径逐步替换

### 6.3 每次只替换一个事实层

- Phase 1 替换原始层接入方式
- Phase 2 替换信息流事实层
- 后面再扩到富化层、聚合层、知识层

这样不会一下把整个项目打散。

## 9. 你现在可以怎么理解它

一句话：

**我们现在确实是在先优化“信息大脑的存储能力”，因为这是后面所有能力的底座。**

只要 Phase 1 和 Phase 2 接好，后面的知识大脑和进化大脑就有了稳定地基。
