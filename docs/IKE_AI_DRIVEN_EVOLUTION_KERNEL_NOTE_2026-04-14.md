# IKE AI-Driven Evolution Kernel Note

Date: 2026-04-14
Status: controller baseline

## Purpose

Prevent project drift toward a merely hard-coded system.

This note exists because recent `Source Intelligence V1` work has necessarily
used bounded heuristics and compression rules, but those rules are not the end
state of IKE.

## Core Judgment

IKE is not being built to become a static rules system.

IKE is being built to become an **AI-driven, continuously improvable system**
whose core remains centered on:

1. information brain
2. knowledge brain
3. evolution brain
4. a scientific methodology layer that cuts across all three

## Non-Forget Rule

The controller must not allow local milestone work to overwrite the project's
main objective.

Current bounded heuristics are allowed only as:

- guardrails
- normalization layers
- noise compression
- temporary truth-preserving scaffolds

They are not allowed to redefine the final architecture as "a hard-coded source
ranking project".

## What Must Remain AI-Driven

The long-term kernel must preserve AI-driven evolution in at least these areas:

### 1. Information Brain

- signal discovery
- source usefulness judgment
- relevance to current task intent
- uncertainty-aware triage

### 2. Knowledge Brain

- reviewed knowledge absorption
- cross-source synthesis
- cross-disciplinary linking
- canonical vs provisional knowledge separation

### 3. Evolution Brain

- prioritization of what to improve next
- diagnosis of system weaknesses
- adaptation of heuristics and policies
- migration from rule-heavy watchdog logic toward model-assisted reasoning

### 4. Scientific Methodology Layer

- hypothesis formation
- experiment packet design
- comparative evaluation
- error analysis
- method learning and iteration

## Current Engineering Interpretation

Recent `M3` ~ `M6` slices should be interpreted as **pre-AI cleanup and
input-quality preparation**, not as the end-state intelligence architecture.

Their role is:

1. remove obvious low-value noise
2. preserve truthful boundaries
3. prepare cleaner inputs for later AI-assisted judgment
4. avoid asking models to reason over avoidable junk

## Guardrail Against Drift

If a work line starts to accumulate too many hard-coded classification or
ranking patches, the controller must ask:

1. is this still only cleanup / normalization?
2. should the next step move into AI-assisted judgment instead?
3. is the rule becoming a substitute for model reasoning?

If the answer to 3 is yes, the line must be compressed or redirected.

## Practical Rule

IKE should use:

- hard rules for safety, contracts, and obvious low-regret cleanup
- AI judgment for semantic value, evolving prioritization, and higher-order
  synthesis

Neither side is sufficient alone.

## Current Mainline Implication

`Source Intelligence V1` should not continue indefinitely as a sequence of
hard-coded patches.

A later slice must explicitly open an **AI-assisted judgment lane** above the
current cleanup foundation.
